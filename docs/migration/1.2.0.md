# Migration guide — `@dashforge/tw@1.2.0`

Sprint 6 closes the Option C rollout (`theme.components.<Name>.defaults`)
across the full catalog and adds 6 new primitives (Link, Select,
Slider, Progress, Stepper, Drawer).

**TL;DR** — this is an **additive release**. No public-API breaking
changes. Every 1.1.x consumer works unchanged after `pnpm add
@dashforge/tw@1.2.0`.

Read this guide if you want to opt into the new features (theme-level
component defaults, the new primitives) or if you're touching
`DashFormBridge` from your own custom form widgets.

---

## Upgrade

```bash
pnpm add @dashforge/tw@1.2.0 @dashforge/tw-theme@1.2.0 @dashforge/tw-tokens@1.2.0
# also bump if you consume the bridge layer:
pnpm add @dashforge/forms@1.0.1 @dashforge/ui-core@1.0.1
```

No code changes required. `pnpm nx run typecheck` should pass with
your current sources.

## What's new

### 6 new primitives

| Component | Purpose | Bridge | Notes |
|---|---|---|---|
| [`<Link>`](https://dashforge-docs-lab.pages.dev/tw/docs/components/link) | Token-driven anchor | No | `startIcon` / `endIcon`, `external`, `asChild` (Radix Slot) |
| [`<Select>`](https://dashforge-docs-lab.pages.dev/tw/docs/components/select) | Enum picker (narrower than Autocomplete) | Yes | Two-arg `onChange(value, option)`, generic `V extends string \| number` |
| [`<Slider>`](https://dashforge-docs-lab.pages.dev/tw/docs/components/slider) | Numeric picker (single OR range) | Yes | `range?: boolean` discriminated union, `commitOnChange` opt-in |
| [`<Progress>`](https://dashforge-docs-lab.pages.dev/tw/docs/components/progress) | Determinate progress (linear + circular) | No | `value` required — Spinner covers indeterminate |
| [`<Stepper>` + `<Step>` + `useStep()`](https://dashforge-docs-lab.pages.dev/tw/docs/components/stepper) | Multi-step wizard | Yes | Compound children, `fields` (RHF trigger) + `isValid` AND semantics, `visibleWhen` per-step |
| [`<Drawer>`](https://dashforge-docs-lab.pages.dev/tw/docs/components/drawer) | Edge-anchored panel | No | 4 positions × 3 variants (`temporary` / `persistent` / `sticky`), resize handle with localStorage persistence |

Each ships as an additive named export from `@dashforge/tw` — no
existing exports were renamed or removed. Import as usual:

```tsx
import { Drawer, Stepper, Step, useStep } from '@dashforge/tw';
```

### Option C — theme-level component defaults

The `theme.components` layer lets you pin DS defaults once, application-wide:

```tsx
import { patchTheme } from '@dashforge/tw-theme';

patchTheme({
  components: {
    // Component-level defaults
    Button: { defaults: { variant: 'solid', color: 'primary', size: 'md' } },
    Chip:   { defaults: { variant: 'outlined', size: 'sm' } },
    Drawer: { defaults: { position: 'right', variant: 'temporary' } },

    // Slot-level defaults (compound components)
    DataGrid: {
      slotProps: {
        header: { className: 'bg-neutral-100 font-semibold' },
        cell:   { className: 'py-3 text-sm' },
      },
    },
  },
});
```

**Precedence chain** (lowest → highest):

1. Tailwind-variants `defaultVariants`
2. `theme.components.<Name>.defaults`
3. Instance props
4. `sx` (via `tailwind-merge`)

Same 4 levels for `slotProps` (theme slotProps → instance slotProps).

**No config = no change**: consumers who don't touch
`theme.components` see identical 1.1.x behavior. The 47 precedence
tests in `libs/dashforge/tw/src/components/*/[A-Z]*.precedence.test.tsx`
verify this baseline explicitly.

Full guide + playground: [Theme system / Component defaults](https://dashforge-docs-lab.pages.dev/tw/docs/theme-system/component-defaults).

### Components explicitly skipped from Option C

Some primitives have no theme-configurable visual axes — Option C is
n/a for them. Documented for clarity:

- **AspectRatio** — `ratio` is an arbitrary number/string, not a
  bounded variant axis
- **VisuallyHidden** — `sr-only`, no visual axes by design
- **Accordion** — Radix behavioral discriminated union (`type`,
  `collapsible`), no identity axes
- **AppShell** — only variant is `navOpen` (behavioral state);
  slotProps candidacy tracked in [#3](https://github.com/kensaadi/dashforge/issues/3)
- **Flex** — was never a primitive; Stack already covers container-level
  flex axes

## Bridge extension — `DashFormBridge.trigger` (optional / non-breaking)

If you build custom form widgets that consume `DashFormContext`
directly, note the new optional method on the bridge:

```ts
interface DashFormBridge {
  // ...existing methods
  trigger?: (name?: string | string[]) => Promise<boolean>;
}
```

- **Consumers that DON'T call `bridge.trigger`** — no change needed.
- **Consumers that build wizard-like flows** — you can now programmatically
  validate a subset of fields on demand:
  ```tsx
  const ok = await bridge.trigger?.(['email', 'phone']);
  if (!ok) {
    // gate the transition, surface errors, etc.
  }
  ```

The `<Stepper>` primitive uses this internally for its `<Step fields={...}>`
validation. The trigger's implementation uses `rhf.getFieldState()` under
the hood so errors are populated synchronously after `await` (working
around a subtle React commit-timing gap).

## Breaking changes — none

None. All Sprint 6 work is additive to the public API.

There are two internal type-narrowing fixes that MAY surface pre-existing
consumer bugs at typecheck time:

### `<Stack gap>` narrower type (Fix [#111](https://github.com/kensaadi/dashforge/issues/111))

Was: `gap?: string | number`. Now: `gap?: 0 | 0.5 | 1 | 2 | 3 | 4 | 6 | 8 | 12 | 16 | 24`.

If your codebase had `<Stack gap="md">` (which was silently accepted
in 1.1.x but produced no CSS), TypeScript now flags it. Fix by
switching to the numeric scale: `<Stack gap={4}>`. A runtime dev-warn
also fires if a non-numeric value reaches the component.

### `sx` no longer clobbers TV variant classes on layout primitives (Fix [#112](https://github.com/kensaadi/dashforge/issues/112))

If you were relying on the buggy `sx` sneak-in behavior on `Box` /
`Stack` / `Grid` / `Container` / `Flex` (where a class in `sx` would
silently overwrite a variant class), the merge order is now
deterministic: variant classes render first, `sx` last, and
`tailwind-merge` picks the winner. Your code likely still works —
`sx="bg-primary-500"` still beats `variant="outlined"` for the
background color. But if you had accidental resets via `sx`, review
those instances.

## Recommended next steps

1. **Adopt `theme.components`** in your app's theme file to remove
   the `sx` boilerplate you have today. Even one component
   (`Button.defaults`) is a solid win. Playground with before/after
   diffs: [Component Defaults Override](https://dashforge-docs-lab.pages.dev/tw/docs/theme-system/component-defaults-override).
2. **Replace bespoke wizards with `<Stepper>`** if you have any.
   Blueprint Builder V2 already migrated — the surface reduction
   was ~200 LOC.
3. **Try `<Drawer variant="sticky">`** for inspector / toolbar
   patterns. Non-dismissible via user gesture, fully programmatic.
4. **Update your custom form-widget code** to use `bridge.trigger`
   for wizard-boundary validation (see the `<Stepper>` source in
   `libs/dashforge/tw/src/components/Stepper/Stepper.tsx` for the
   canonical pattern).
