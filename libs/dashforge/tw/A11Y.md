# A11Y audit — `@dashforge/tw`

Per-component accessibility status for every shipped `@dashforge/tw`
component (24 total). Audit date: 2026-05-17, against the 0.2.0-beta
codebase (+ the three pending lib fixes for Checkbox / RadioGroup /
NumberField + the Button `aria-busy` improvement landing in 0.2.1-beta).

Reference: [WCAG 2.1 AA](https://www.w3.org/TR/WCAG21/) +
[WAI-ARIA Authoring Practices](https://www.w3.org/WAI/ARIA/apg/).

## Summary

| Group | Components | Status |
|---|---|---|
| Foundation (presentational) | Typography, Box, Stack, Grid, Container, AspectRatio | ✅ N/A — pass-through to HTML semantics |
| Foundation (a11y primitive)  | VisuallyHidden, Divider                              | ✅ Conformant |
| Form controls (Radix-backed) | Checkbox, Switch, RadioGroup                         | ✅ Inherits WAI-ARIA APG from Radix |
| Form controls (custom)       | Button, TextField, Textarea, NumberField, OTPField, Autocomplete, DateTimePicker | ✅ Conformant (see per-row notes) |
| Layout / Navigation          | AppShell, TopBar, LeftNav, Breadcrumbs               | ✅ Conformant |
| Overlays (Radix-backed)      | ConfirmDialog                                        | ✅ Inherits from Radix Dialog |
| Notifications                | Snackbar                                             | ✅ Conformant (aria-live polite) |

Zero blocking failures. One enhancement landed: Button now exposes
`aria-busy={loading}` so assistive tech distinguishes loading from
plain disabled.

## Per-component findings

### Foundation

| Component | Status | Notes |
|---|---|---|
| **Typography**     | ✅ | Renders the variant-appropriate HTML tag (`<h1>`–`<h6>`, `<p>`, `<small>`) so screen-reader heading navigation works out of the box. `as` / `asChild` let consumers re-tag without losing styles. |
| **Box**            | ✅ | Pass-through `<div>` (or `as`/`asChild`-overridden tag). No interactive semantics. |
| **Stack**          | ✅ | Pass-through `<div>`. Display-only flex container. |
| **Grid**           | ✅ | Pass-through `<div>` with `display: grid`. Layout-only. |
| **Container**      | ✅ | Pass-through `<div>` or `<main>` (via `asChild`). Layout-only. |
| **Divider**        | ✅ | Horizontal: `<hr role="separator" aria-orientation="horizontal">`. Vertical: `<div role="separator" aria-orientation="vertical">`. Labeled variant keeps the role on the wrapper + decorative line segments carry `aria-hidden`. |
| **AspectRatio**    | ✅ | Pass-through `<div>` carrying a `style={aspect-ratio: ...}`. No interactive semantics. |
| **VisuallyHidden** | ✅ | IS the a11y primitive — WebAIM clip technique via Tailwind's `sr-only`. Chooses `<span>` by default (works inside buttons / links without breaking inline flow). |

### Tier-1 form controls

| Component   | Status | Notes |
|-------------|--------|-------|
| **Button**     | ✅ | Native `<button>` (or Slot via `asChild`). `disabled` + `aria-disabled` mirrored. **`aria-busy={true}` while loading** (added 2026-05-17) — SR users hear "busy" instead of just "dimmed". Loading spinner is `aria-hidden`. `focus-visible:ring-*` matches the variant color. |
| **TextField**  | ✅ | `<label htmlFor>` via `useId()`. `aria-invalid` on error, `aria-describedby` linked to the helper-text id. Required marker is `aria-hidden` decorative + the real signal is the native `required` attribute. `placeholder:text-neutral-400`. |
| **Checkbox**   | ✅ | Radix `Checkbox.Root` (WAI-ARIA APG). Indicator now uses Radix-native mounting (no React conditional) so checked-state changes from any source — click, programmatic, bridge — propagate to AT. |
| **Switch**     | ✅ | Radix `Switch.Root` (WAI-ARIA APG). `role="switch"` + `aria-checked`. Thumb animation gated on `data-state` — pure CSS, no React state for the visual. |

### Tier-2 / 3 form controls

| Component         | Status | Notes |
|-------------------|--------|-------|
| **RadioGroup**       | ✅ | Radix `RadioGroup.Root` + `Item`. `role="radiogroup"` on the container, `role="radio"` + `aria-checked` per option. Group label linked via `aria-labelledby`. Per-option RBAC hide preserves the selected option as disabled (so the user can see what they picked even after a perm change). |
| **Textarea**         | ✅ | Same label + helper-text wiring as TextField. `<textarea>` native element, `resize` axis controlled via CSS. |
| **NumberField**      | ✅ | Native `<input type="number">` — keyboard arrow keys are the accessible way to change the value (browser default). The custom +/− stepper is **intentionally `aria-hidden`** and `tabIndex={-1}` — visible/touch affordance only, with `aria-label="Increment"` / `"Decrement"` for any AT that does walk into hidden subtrees. |
| **OTPField**         | ✅ | One real `<input>` (visually rendered as N segmented slots via decorative `aria-hidden` `<div>`s). `autoComplete="one-time-code"` for SMS autofill on iOS. `inputMode="numeric"` (default) for mobile keyboards. Backspace / Delete / paste all work natively because the underlying element IS a single input. |
| **Autocomplete**     | ✅ | Full WAI-ARIA Combobox pattern: `role="combobox"` + `aria-autocomplete="list"` + `aria-expanded` + `aria-controls` + `aria-activedescendant`. Listbox: `role="listbox"` + per-option `role="option"` + `aria-selected`. Async load row: `aria-live="polite"` + `aria-busy="true"`. Clear / open buttons carry `aria-label`. Chip remove buttons: `aria-label="Remove {label}"`. |
| **DateTimePicker**   | ✅ | Native HTML5 `<input type="date">` / `"time"` / `"datetime-local"`. Browser ships a fully-accessible picker for each. `aria-invalid` + `aria-describedby` wired to the helper text. |

### Layout / navigation

| Component     | Status | Notes |
|---------------|--------|-------|
| **AppShell**     | ✅ | `<aside>` for desktop nav, `<aside aria-hidden={!navOpen}>` for the mobile drawer copy (so it doesn't double-announce when hidden). Escape closes the drawer. `<main>` landmark wraps content. `<header>` / `<footer>` are native landmarks. **Known limitation**: no focus trap inside the mobile drawer — focus returns to the toggle button on Escape close, which is acceptable for a docs site / typical web app but not full WAI-ARIA modal-drawer compliance. Can be added in a future revision. |
| **TopBar**       | ✅ | Renders as `<header>` (banner landmark) by default. `asDiv` opt-out for nested layouts where a banner landmark would duplicate. |
| **LeftNav**      | ✅ | `<nav aria-label>` landmark. `aria-current="page"` on the active row. Groups use `aria-expanded` + `aria-controls` linking the header `<button>` to a `role="region" aria-labelledby` panel. Collapse toggle: `aria-pressed` + dynamic `aria-label`. Per-row RBAC hide is matched by an a11y-tree removal (not just visual), so the announced nav matches what the user can actually interact with. |
| **Breadcrumbs**  | ✅ | `<nav aria-label="Breadcrumb">` + `<ol>` (screen readers announce position-in-list). `aria-current="page"` on the last (or `current: true`) crumb. Separators carry `aria-hidden="true"`. Truncation ellipsis: `<span aria-label="More breadcrumbs">`. |

### Overlays + notifications

| Component        | Status | Notes |
|------------------|--------|-------|
| **ConfirmDialog**   | ✅ | Radix `Dialog.Root` (WAI-ARIA APG). `role="dialog"` + `aria-modal="true"` + `aria-labelledby` on the title. Radix handles focus trap, focus restoration on close, Escape close, scroll lock. |
| **Snackbar**        | ✅ | Stack is `role="region" aria-label="Notifications" aria-live="polite" aria-atomic="false"`. Polite (not assertive) so notifications don't interrupt the user's screen-reader stream; `atomic="false"` so only the changed item is announced, not the whole queue. Per-item dismiss button: `aria-label="Dismiss notification"`. |

## Cross-cutting concerns

- **Focus visibility** — every interactive component carries
  `focus-visible:ring-*` matching its color variant. `outline-none` is
  always paired with a ring so keyboard users never lose the focus
  indicator (regression-tested via the inventory grep:
  `Button / Checkbox / ConfirmDialog / RadioGroup / Switch / TextField
  / Textarea / NumberField / OTPField / Autocomplete / DateTimePicker
  / Breadcrumbs / LeftNav / Snackbar` all have `focus-visible:ring`).
- **Color contrast** — the design tokens emit foreground/background
  pairs that meet WCAG 2.1 AA (4.5:1 for normal text, 3:1 for large
  text). Independent verification is on the dashforge-tokens roadmap.
- **Reduced motion** — animations (Switch thumb slide, Snackbar slide-in,
  ConfirmDialog fade) use the default `transition-*` utilities. A
  `prefers-reduced-motion` audit pass is filed as a future enhancement
  (current behavior: animations always play; not a WCAG failure but
  reduces polish for users with motion sensitivity).
- **Form error association** — every form control with `helperText` +
  `error` wires `aria-invalid` and `aria-describedby` to the helper id
  so the error message is announced alongside the field name.

## Known limitations & future work

1. **AppShell mobile drawer focus trap** — currently relies on Escape
   to close + focus naturally returning to the toggle button. A future
   release can add a real focus trap (probably via `focus-trap-react`
   or hand-rolled).
2. **`prefers-reduced-motion` audit** — animations always play. Should
   gate the Switch thumb slide, Snackbar slide-in, ConfirmDialog fade
   on the media query.
3. **High-contrast Windows mode** — not explicitly tested. The `border`
   utility renders a 1px border that should survive forced-colors mode,
   but full validation pending.
4. **Color contrast regression suite** — should be added to CI so a
   token recolor doesn't silently regress AA compliance.

## Audit methodology

- ARIA role / attribute inventory via `grep -nE "aria-[a-z]+=|role=|onKeyDown"`
  per component source.
- Keyboard handler audit by reading each component's `onKeyDown`,
  `onPaste`, `onFocus`, `onBlur` and tracing into the registered
  bridge / Radix primitive.
- Focus visibility inventory via `grep "focus-visible:ring"` across
  all `*.variants.ts`.
- Label association verification: `useId()` → `htmlFor=inputId` +
  `aria-describedby=helperId` chain checked for TextField, Textarea,
  NumberField, OTPField, DateTimePicker.

No automated lighthouse / axe scan was run as part of this audit —
that's a separate CI integration on the roadmap. Manual screen-reader
verification (NVDA on Windows, VoiceOver on macOS / iOS) should be
periodically refreshed; results land in this file as they happen.
